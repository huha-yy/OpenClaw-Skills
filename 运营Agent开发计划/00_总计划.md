# 运营 Agent 开发总计划

## 1. 项目定位

本项目目标是基于 OpenClaw / openclawd 搭建一套面向运营团队的自媒体运营 Agent 工作流。

整体建设分两大阶段推进：

```text
第一阶段：热点内容生产工作流
目标：从热点监控到三平台发布包输出，先解决内容生产效率问题。

第二阶段：自动化运营发布工作流
目标：在第一阶段发布包和审核机制稳定后，逐步接入平台发布、草稿、评论管理、数据回收等自动化运营能力。
```

也就是说，第一阶段不是最终形态，而是为第二阶段自动化发布和运营打基础。

## 2. 总体路线

两大阶段，每个阶段包含若干子阶段：

```text
第一阶段：热点内容生产工作流
  ├─ 子阶段 0：运营需求确认
  ├─ 子阶段 1：热点内容生产 POC
  ├─ 子阶段 2：Agent 工作流成型
  └─ 子阶段 3：发布包与审核机制
        ↓
第二阶段：自动化运营发布工作流
  ├─ 子阶段 4：自动化运营发布
  ├─ 子阶段 5：数据回收与复盘优化
  └─ 子阶段 6：公司环境部署与多账号扩展
```

## 3. 总体工作流

最终工作流目标：

```text
热点监控
  ↓
公司相关性判断
  ↓
多源事实摘要
  ↓
内容策略生成
  ↓
三平台内容包生成
  ↓
风险审核
  ↓
人工确认 / 自动规则判断
  ↓
自动草稿 / 自动发布 / 发布包导出
  ↓
评论收集 / 回复建议 / 低风险自动回复
  ↓
数据回收
  ↓
复盘与下一轮选题优化
```

## 4. 第一阶段：热点内容生产工作流

> 目标：从热点监控到三平台发布包输出，先解决内容生产效率问题。

### 4.1 子阶段 0：运营需求确认

#### 目标

确认运营真实需求、业务目标、平台优先级、审核边界和自动化接受程度。

#### 输出

- 运营需求确认记录
- 第一阶段内容生产需求
- 第二阶段自动化发布需求
- 审核与风险边界
- 平台优先级
- 成功标准

#### 重点确认

```text
1. 运营最需要自动化的是哪一段？
2. 第一阶段是否接受发布包 + 人工审核？
3. 第二阶段哪些平台需要自动发布？
4. 哪些内容可以自动发布？哪些必须人工审核？
5. 自动评论/点赞/转发背后的真实业务目的是什么？
```

### 4.2 子阶段 1：热点内容生产 POC

#### 目标

跑通最小内容生产闭环：

```text
一个热点输入 → 相关性判断 → 多源摘要 → 三平台内容包 → 风险报告 → 发布包输出
```

#### 第一阶段输出

```text

    event_analysis.md
    risk_report.md
    package_summary.md

    xiaohongshu/
      note.md

      image_cards.json
      publish_meta.json

    douyin/
      video_script.md
      storyboard.md
      subtitles.srt
      cover.md
      publish_meta.json

    wechat/
      article.md
      cover.md
      inline_images.md
      layout.md
      publish_meta.json
```

#### 第一阶段重点

- 热点输入
- 关键词匹配
- 相关性判断
- 多源事实摘要
- 内容策略
- 三平台内容包
- 风险提示
- 发布包输出

#### 第一阶段不重点做

第一阶段暂不直接做真实平台发布，重点验证内容质量和运营可用性。

### 4.3 子阶段 2：Agent 工作流成型

将第一阶段的流程拆分成可维护、可编排、可扩展的 Agent 和工具函数。

| 2 | Relevance Agent | 判断公司相关性 | `skills/relevance-judge/SKILL.md` | 已就绪 |
| 3 | Research Agent | 整理来源和事实点 | `skills/fact-research/SKILL.md` | 已就绪 |
| 4 | Strategy Agent | 生成内容角度 | `skills/content-strategy/SKILL.md` | 已就绪 |


### 4.4 子阶段 3：发布包与审核机制

#### 目标

让运营能基于发布包完成审核、修改、确认发布。

#### 审核状态

```text
待审核
审核通过
审核驳回
需补充事实
需品牌复核
已发布
发布失败
```

#### 审核重点

- 事实来源
- 品牌口径
- 平台风格
- 版权风险
- 敏感词风险
- 高风险热点识别

## 5. 第二阶段：自动化运营发布工作流

> 目标是逐步实现从”发布包输出”到”自动化运营发布”的升级。不是一上来全平台无人值守发布，而是分层推进。

### 5.1 子阶段 4：自动化运营发布

```text
发布包生成
  ↓
自动创建草稿
  ↓
人工审核确认发布
  ↓
低风险内容自动排期发布
  ↓
发布状态回收
  ↓
评论收集与回复建议
  ↓
数据回收与复盘
```

#### 发布自动化分级

| 等级 | 名称 | 说明 | 适用阶段 |
| --- | --- | --- | --- |
| L0 | 发布包导出 | 只生成内容和素材，人工发布 | 第一阶段 |
| L1 | 自动创建草稿 | 系统把内容写入平台草稿箱 | 第二阶段初期 |
| L2 | 人工确认发布 | 系统准备发布任务，人工点击确认 | 第二阶段核心 |
| L3 | 低风险自动排期发布 | 低风险内容按规则自动发布 | 第二阶段后期 |
| L4 | 全自动发布 | 无人审核直接发布 | 暂不作为默认目标 |

第一版到第二阶段的推荐目标是 L1-L2，后续再评估 L3。

#### 平台发布策略

| 平台 | 第二阶段策略 | 说明 |
| --- | --- | --- |
| 微信公众号 | 优先接草稿/发布接口 | 官方能力相对明确，适合优先接入 |
| 小红书 | 先发布包，后评估官方/合规发布能力 | 不建议第一时间模拟 App 发布 |
| 抖音 | 先发布包，后评估开放平台能力 | 视频发布需考虑素材、封面、话题和账号权限 |

#### 自动化发布工作流

```text
发布包生成
  ↓
风险等级判断
  ↓
审核规则判断
  ↓
生成发布任务
  ↓
平台 Connector 转换格式
  ↓
创建草稿 / 排期任务
  ↓
运营确认
  ↓
执行发布
  ↓
记录发布结果
  ↓
失败重试 / 人工提醒
```

#### 发布任务状态

```text
created              已创建
waiting_review       待审核
approved             已通过
rejected             已驳回
draft_created        草稿已创建
scheduled            已排期
publishing           发布中
published            已发布
failed               发布失败
cancelled            已取消
```

#### 自动化运营互动

第二阶段也需要考虑评论区和互动，但需要分层推进。

推荐顺序：

```text
1. 评论收集
2. 评论分类
3. 高风险评论识别
4. 回复建议生成
5. 人工确认回复
6. 常见问题低风险自动回复
7. 点赞/转发等动作根据平台规则再评估
```

不建议第二阶段一开始就做：

```text
批量自动点赞
批量自动转发
批量自动评论
模拟真人互动
多账号刷互动
```

#### 第二阶段核心交付物

- 平台发布 Connector 设计
- 发布任务状态机
- 自动草稿能力
- 人工确认发布流程
- 发布失败重试机制
- 评论收集和回复建议
- 发布结果日志
- 简单数据回收

### 5.2 子阶段 5：数据回收与复盘优化

#### 目标

将发布后的数据回收到系统中，形成内容优化闭环。

#### 数据指标

```text
阅读量
播放量
点赞量
收藏量
评论量
转发量
涨粉量
私信量
线索量
完播率
点击率
```

#### 输出

- 单条内容效果报告
- 平台效果对比
- 热点跟进效果分析
- 标题和内容风格建议
- 下一轮选题建议

### 5.3 子阶段 6：公司环境部署与多账号扩展

#### 目标

将本地 POC 迁移到公司环境，并支持后续多账号、多平台、多业务线运营。

#### 公司环境能力

- 数据库存储
- 对象存储
- 任务调度
- 密钥管理
- 权限管理
- 审核流
- 日志审计
- 告警通知
- 多账号隔离

```text

热点内容生产 POC
三平台内容包
风险审核
发布包输出
```

### P1：第二阶段重点推进

```text
自动创建草稿
人工确认发布
发布任务状态机
发布结果记录
评论收集
回复建议
简单数据回收
```

### P2：后续增强

```text

```

### P3：谨慎评估

```text

```

## 7. 风险控制原则

即使进入第二阶段自动化发布，也需要遵守以下原则：

```text
1. 优先使用官方接口或合规能力。
2. 高风险内容必须人工审核。
3. 平台账号操作必须可追溯。
4. 自动发布必须有频率限制。
5. 发布失败必须可重试、可回滚、可提醒人工。
6. 评论自动回复必须限制在低风险 FAQ 场景。
7. 自动点赞、转发、批量评论需要单独评估平台风险。
```


## 9. 当前建议结论

与两大阶段对齐：

```text
第一阶段：先跑通内容生产和发布包。
第二阶段：再做自动草稿、人工确认发布、低风险排期发布、评论回复建议、
           数据回收、复盘优化和多账号扩展。
```

这样既能覆盖运营提出的”自动化运营发布”诉求，又能避免第一版直接陷入平台发布、账号风控和互动合规风险。
