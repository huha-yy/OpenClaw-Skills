# 技术栈选型文档

## 1. 文档目的

说明运营 Agent 项目在两个阶段中的技术栈选择、备选方案和推荐路线。

```text
第一阶段：热点内容生产工作流
  目标：热点监控 → 相关性判断 → 多源摘要 → 三平台内容包 → 发布包输出

第二阶段：自动化运营发布工作流
  目标：接入草稿、发布任务、人工确认、低风险排期发布、评论收集和数据回收
```

## 2. 总体技术路线

```text
OpenClaw 作为主编排中枢
本地工具函数作为业务执行层
配置文件管理关键词、热点源和品牌规范
第一阶段用文件系统输出发布包
第二阶段引入发布任务状态、平台 Connector、审核流和数据回收
```

不建议第一版同时引入过多框架（如 OpenClaw + n8n + Dify + Coze + LangChain + LangGraph），技术栈越重越拖慢落地。

## 3. 第一阶段技术栈

### 3.1 推荐技术栈

| 模块       | 推荐                       | 说明                          |
| ---------- | -------------------------- | ----------------------------- |
| Agent 编排 | OpenClaw                   | 核心 Agent 调度和模型调用入口 |
| 模型调用   | 小米 MiMo via OpenClaw     | 当前本地已配置可用            |
| 工具函数   | Python/Node.js 脚本        | 热点拉取、文件读写、格式转换  |
| 配置管理   | YAML/JSON/Markdown         | 管理关键词、热点源、品牌规范  |
| 输出存储   | 本地文件系统               | 输出标准发布包到 `outputs/`   |
| 日志       | 本地日志文件               | 记录执行过程和失败原因        |
| 图片策略   | Pexels/图库检索/SD/ComfyUI | 详见图片策略文档              |
| 审核方式   | 人工查看发布包             | 第一阶段不做复杂审核页面      |

### 3.2 为什么暂不引入以下框架

| 框架      | 原因                                                         | 何时引入                         |
| --------- | ------------------------------------------------------------ | -------------------------------- |
| n8n       | 第一阶段流程线性，无需可视化编排；引入会增加部署维护成本     | 第二阶段如需可视化审批/通知      |
| Dify      | 与 OpenClaw Agent 编排目标重叠，对复杂平台自动化不如自研灵活 | 除非公司已有 Dify 平台           |
| Coze      | 偏 Bot 和低代码 Agent，不適合作为核心工程底座                | 不作主技术栈                     |
| LangChain | 第一阶段不是必要依赖，与 OpenClaw 工具能力部分重叠           | 需要 RAG/文档检索时局部引入      |
| LangGraph | 第一阶段无复杂状态机、人审、暂停恢复需求                     | 第二阶段发布状态机复杂后局部引入 |

第一阶段流程是线性的 `A → B → C → D → E`，OpenClaw + 本地工具函数即可完成。

## 4. 第二阶段技术栈

### 4.1 推荐技术栈

| 模块           | 推荐                         | 说明                               |
| -------------- | ---------------------------- | ---------------------------------- |
| 主编排         | OpenClaw/openclawd           | 仍作为核心编排入口                 |
| 发布任务状态   | SQLite → PostgreSQL/MySQL    | 本地先 SQLite，公司环境升级        |
| 平台 Connector | 自研 Connector               | 对接微信公众号、抖音、小红书等接口 |
| 定时任务       | OpenClaw cron / 公司调度系统 | 定时监控、排期发布、数据回收       |
| 审核流         | 轻量页面/公司内部系统        | 待审核、通过、驳回、已发布状态     |
| 通知           | 飞书/企业微信/邮件           | 可通过自研或 n8n 接入              |
| 评论处理       | OpenClaw Agent + Connector   | 收集评论、分类、生成回复建议       |
| 数据回收       | Connector + 数据库           | 回收阅读、播放、点赞、评论等指标   |

### 4.2 可选增强

| 组件      | 用途                               | 条件                                     |
| --------- | ---------------------------------- | ---------------------------------------- |
| LangGraph | 复杂发布状态机、人审流程、失败重试 | 当状态流复杂度明显超过 OpenClaw 舒适区时 |
| n8n       | 可视化编排、第三方 SaaS 集成、通知 | 当需要运营人员可视化配置流程时           |
| LangChain | RAG、文档加载、向量检索、知识库    | 当需要构建行业知识库或增强事实核验时     |

## 5. 备选技术评估（精简）

### n8n

- **适合：** 可视化流程、定时触发、Webhook、飞书/企微通知、审批流
- **不适合：** 作为核心编排引擎（与 OpenClaw 职责重叠）
- **建议：** 第一阶段不用，第二阶段如公司已有 n8n 或需要可视化审批，可作为外层流程工具

### Dify

- **适合：** 快速搭建 LLM 应用、知识库问答、简单工作流
- **不适合：** 复杂平台自动化、发布状态机、公司内部部署约束
- **建议：** 除非公司已有 Dify 平台，否则不优先引入

### Coze

- **适合：** 快速搭 Bot、轻量自动问答
- **不适合：** 自定义工作流、平台发布状态、公司环境集成
- **建议：** 不作为主技术栈

### LangChain

- **适合：** RAG、文档处理、工具封装、知识库
- **不适合：** 作为主编排框架
- **建议：** 需要 RAG 或文档检索时局部引入，不作为主编排

### LangGraph

- **适合：** 有状态、多分支、可暂停恢复的人机协作工作流
- **不适合：** 第一阶段线性流程
- **建议：** 第一阶段不用，第二阶段发布任务复杂后作为局部状态机引入

## 6. 推荐架构

### 第一阶段

```text
OpenClaw Gateway
  ↓
OpenClaw Agent 工作流
  ↓
本地工具函数 (Python/Node.js)
  ↓
配置文件 (YAML/JSON/Markdown)
  ↓
发布包输出到 outputs/
```

### 第二阶段

```text
OpenClaw
  ↓
Agent 工作流
  ↓
发布任务状态机（可选 LangGraph）
  ↓
平台 Connector（自研）
  ↓
草稿/发布/评论/数据接口
  ↓
审核流 + 日志 + 通知（可选 n8n）
```

## 7. 最终推荐

```text
第一阶段：OpenClaw + 本地工具函数 + 文件输出
  不引入 n8n / Dify / Coze / LangGraph
  LangChain 仅在 RAG 需求明确时局部引入

第二阶段：OpenClaw + 平台 Connector + 发布任务状态 + 审核流
  可选增强：LangGraph（复杂状态流）、n8n（可视化编排）、LangChain（RAG/文档处理）
```
