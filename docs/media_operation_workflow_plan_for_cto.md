# 自媒体热点运营 Agent 工作流计划

## 1. 一句话结论

第一版不做“全自动账号运营机器人”，而是先完成一套可落地、可审核、可迁移的自媒体热点运营工作流：

```text
热点监控 → 相关性判断 → 多源核验 → 内容策略 → 三平台图文/视频内容包 → 风险审核 → 人工确认 → 发布包输出
```

第一版核心价值是提高运营从“发现热点”到“形成可发布内容草稿”的效率，优先解决选题、资料整理、多平台改写和图文协同内容准备的问题。

## 2. 项目背景

当前运营侧提出的原始需求为：

```text
1. 自动便捷管理自媒体，包括不限于抖音、小红书、微信公众号等。
2. 要求能实现自动发布内容、视频等，自动评论点赞转发。
```

经过初步分析，该需求不应直接理解为“自动发帖、自动刷互动”，而应拆解为以下真实业务诉求：

- 更快发现与公司、行业、产品、竞品相关的热点。
- 更快判断热点是否值得跟进。
- 更快产出适配不同平台的内容。
- 降低运营在选题、资料收集、改写、排版、发布准备上的重复劳动。
- 在保证品牌口径和事实准确的前提下，形成可审核、可追溯的内容生产流程。

因此，第一版建议聚焦“热点驱动的内容生产工作流”，而不是高风险的全自动互动操作。

## 3. 第一版目标

### 3.1 核心目标

第一版需要跑通一个完整闭环：

```text
输入：热点源、公司关键词、品牌资料、历史内容参考
处理：发现热点、判断相关性、核验事实、生成内容、输出风险提示
输出：小红书图文内容包、抖音视频内容包、微信公众号文章内容包、发布包汇总
```

### 3.2 业务目标

- 帮助运营快速发现可借势热点。
- 将热点资料整理、摘要、改写、多平台适配自动化。
- 将单条热点从“发现到初稿”的时间压缩到可控范围内。
- 保留人工审核环节，降低事实、品牌、合规风险。
- 为后续接入自动草稿、自动发布、数据复盘打基础。

### 3.3 第一版成功标准

建议以以下指标衡量第一版是否有效：

- 每次运行可输出候选热点列表。
- 对命中的热点能给出相关性说明和推荐跟进理由。
- 对每个可跟进热点，能生成三平台内容包。
- 内容包包含文字、图文结构、封面建议、视频脚本、风险提示和来源链接。
- 运营可基于输出内容进行人工审核、修改和发布。
- 全流程有文件输出和日志记录，便于复盘。

## 4. 第一版范围

### 4.1 P0：必须实现

P0 是第一版必须完成的主线能力。

| 模块 | 说明 | 输出 |
| --- | --- | --- |
| 热点监控 | 从指定热点源/新闻源拉取候选热点 | 候选热点列表 |
| 关键词匹配 | 根据公司、产品、行业、竞品关键词筛选热点 | 初筛结果 |
| 相关性判断 | 判断热点与公司/行业是否相关 | 相关性评分与原因 |
| 多源摘要 | 拉取并整理 2-3 个来源的摘要和事实点 | 来源与事实摘要 |
| 内容策略 | 判断适合做什么角度的内容 | 推荐选题角度 |
| 小红书内容包 | 输出图文协同笔记方案 | 标题、正文、图卡结构、标签 |
| 抖音内容包 | 输出短视频内容方案 | 标题、口播、分镜、字幕、封面 |
| 公众号内容包 | 输出长文内容方案 | 标题、摘要、正文、插图建议 |
| 风险提示 | 标记事实、品牌、平台、合规风险 | 风险报告 |
| 发布包输出 | 将所有内容按标准目录输出 | 可审核发布包 |

### 4.2 P1：第二优先级

P1 能提升使用体验，但不阻塞第一版闭环。

- 本地审核页面。
- 运营手动选择热点后再生成内容。
- 支持历史爆款内容作为风格参考。
- 支持品牌语气、禁用词、敏感词配置。
- 支持公众号草稿箱接口。
- 支持内容版本管理。
- 支持简单日报/周报。

### 4.3 P2：后续增强

P2 属于后续阶段，不建议第一版投入。

- 抖音、小红书自动发布。
- 自动评论、点赞、转发。
- 多账号矩阵分发。
- 自动生成完整视频成片。
- 自动生成图片并直接发布。
- 舆情告警和负面处理工作台。
- 完整数据看板和投放归因。

## 5. 明确暂不做范围

为避免第一版失控，以下能力不纳入第一版：

```text
1. 不做无人值守自动发布到抖音/小红书。
2. 不做自动点赞、收藏、转发、批量评论。
3. 不做绕过平台风控、模拟 App、Cookie 登录等非官方操作。
4. 不做高风险热点的自动发布。
5. 不做完整视频自动剪辑成片，只做脚本、分镜、字幕和素材建议。
6. 不做正式生产环境多账号矩阵管理。
```

第一版输出的是“可审核发布包”，不是“自动发布结果”。

## 6. 工作流设计

### 6.1 总体流程

```text
1. 配置热点源和公司关键词
        ↓
2. 定时拉取热点/新闻
        ↓
3. 去重与聚类
        ↓
4. 判断与公司/行业/竞品的相关性
        ↓
5. 选择值得跟进的热点
        ↓
6. 拉取多个来源并提取事实点
        ↓
7. 生成内容策略和平台角度
        ↓
8. 生成小红书图文内容包
        ↓
9. 生成抖音视频内容包
        ↓
10. 生成微信公众号文章内容包
        ↓
11. 生成风险提示和审核建议
        ↓
12. 输出标准发布包
        ↓
13. 运营人工审核、修改、发布
```

### 6.2 工作流主线说明

第一版主线不是“平台账号控制”，而是“热点到内容包”的生产流程。

核心输入：

```text
- 公司关键词
- 品牌资料
- 产品资料
- 竞品关键词
- 热点源配置
- 历史内容参考
```

核心输出：

```text
- 热点摘要
- 相关性判断
- 内容角度建议
- 小红书图文内容包
- 抖音视频内容包
- 微信公众号文章内容包
- 风险报告
- 发布包汇总
```

## 7. 发布包定义

发布包是系统交付给运营审核和发布的一组标准化文件。它不是已经发布的内容，而是“待审核、待发布的内容交付包”。

### 7.1 推荐目录结构

```text
outputs/
  2026-06-09_热点标题/
    sources.json
    event_analysis.md
    risk_report.md
    package_summary.md

    xiaohongshu/
      note.md
      image_storyboard.md
      image_cards.json
      publish_meta.json

    douyin/
      video_script.md
      storyboard.md
      subtitles.srt
      cover.md
      publish_meta.json

    wechat/
      article.md
      cover.md
      inline_images.md
      layout.md
      publish_meta.json
```

### 7.2 小红书图文内容包

小红书不能只输出正文，应输出图文协同内容。

内容包括：

- 笔记标题
- 正文
- 标签
- 评论区引导语
- 封面图文字
- 每张图的主题
- 每张图的图上文字
- 每张图的画面说明
- 每张图与正文段落的对应关系

示例结构：

```json
{
  "platform": "xiaohongshu",
  "title": "新能源补贴又变了？买车前先看这 3 点",
  "images": [
    {
      "index": 1,
      "type": "cover",
      "text_on_image": "新能源补贴又变了？",
      "visual": "新能源车和政策标签的生活化场景",
      "purpose": "吸引点击，说明主题"
    },
    {
      "index": 2,
      "type": "key_points",
      "text_on_image": "重点看这 3 点",
      "visual": "三栏信息卡片：地区、时间、车型条件",
      "purpose": "承接正文核心观点"
    }
  ]
}
```

### 7.3 抖音视频内容包

抖音不是只输出脚本，应输出短视频制作所需内容。

内容包括：

- 视频标题
- 口播脚本
- 分镜脚本
- 字幕文本
- 封面标题
- 话题标签
- 画面素材建议
- 发布时间建议

### 7.4 微信公众号内容包

公众号不是只输出长文，应输出文章与图文排版方案。

内容包括：

- 文章标题
- 摘要
- 正文
- 小标题
- 封面图建议
- 正文插图建议
- 插图位置
- 引用来源
- 风险提示

## 8. Agent 分工设计

OpenClaw 在该方案中作为 Agent 编排中枢，各 Agent 分工如下：

| Agent | 职责 | 第一版优先级 |
| --- | --- | --- |
| Monitor Agent | 拉取热点源、新闻源、行业源 | P0 |
| Relevance Agent | 判断热点与公司/行业/竞品的相关性 | P0 |
| Research Agent | 拉取来源、提取事实点、多源摘要 | P0 |
| Strategy Agent | 判断内容角度、平台适配策略 | P0 |
| XHS Writer Agent | 生成小红书图文内容包 | P0 |
| Douyin Writer Agent | 生成抖音视频内容包 | P0 |
| WeChat Writer Agent | 生成公众号文章内容包 | P0 |
| Compliance Agent | 检查事实、品牌、合规和平台风险 | P0 |
| Package Agent | 汇总标准发布包 | P0 |
| Publisher Agent | 调用平台发布/草稿接口 | P1/P2 |
| Analyst Agent | 数据复盘与优化建议 | P1/P2 |

## 9. 本地 POC 计划

### 9.1 本地 POC 目标

在本地环境先跑通最小闭环，验证业务逻辑和输出质量。

```text
输入：热点源 + 公司关键词 + 品牌资料
输出：一个热点对应的三平台发布包
```

### 9.2 本地 POC 技术形态

建议第一版采用轻量方式：

- 本地配置文件管理关键词和热点源。
- 本地脚本执行热点拉取与内容生成。
- OpenClaw 负责 Agent 编排和模型调用。
- 文件夹输出发布包。
- 人工查看 Markdown/JSON 结果。
- 暂不依赖复杂前端和数据库。

## 10. 阶段计划

### 阶段 0：运营需求确认

目标：确认第一版真实需求和边界。

交付物：

- 运营侧需求确认文档
- 第一版范围说明
- 不做范围说明
- 样例输出格式

重点确认：

```text
1. 运营最关注的平台优先级。
2. 需要监控的热点类型和关键词。
3. 第一版是否接受发布包 + 人工审核。
4. 评论/点赞/转发背后的真实目的。
5. 内容风格和品牌禁用表达。
```

### 阶段 1：本地最小闭环

目标：跑通“热点到发布包”的单条链路。

交付物：

- 热点源配置
- 公司关键词配置
- 热点拉取 demo
- 相关性判断结果
- 三平台内容包
- 风险报告
- 标准输出目录

验收标准：

```text
输入一个热点或从热点源拉取一个热点后，系统能生成完整发布包。
```

### 阶段 2：批量热点处理

目标：支持定时拉取、筛选、排序多个热点。

交付物：

- 热点列表
- 去重聚类
- 相关性评分
- 推荐跟进列表
- 批量生成发布包

验收标准：

```text
系统能从多个热点中筛出值得运营关注的内容，并按优先级输出。
```

### 阶段 3：审核与运营体验

目标：让运营能更方便地查看、筛选、修改和确认内容。

交付物：

- 本地审核页面或简易工作台
- 内容状态：待审核/已通过/已驳回/已发布
- 修改记录
- 审核意见

验收标准：

```text
运营可以基于系统输出完成审核和发布准备，不需要直接读大量原始文件。
```

### 阶段 4：公司环境试部署

目标：将本地 POC 迁移到公司测试环境。

交付物：

- 任务调度
- 数据库存储
- 文件/对象存储
- 日志审计
- 权限控制
- 密钥管理

验收标准：

```text
公司测试环境中可稳定运行热点监控和内容包生成流程。
```

## 11. 风险与处理策略

| 风险 | 说明 | 处理策略 |
| --- | --- | --- |
| 平台合规风险 | 自动点赞、评论、转发、模拟发布可能触发平台风控 | 第一版不做高风险互动和无人发布 |
| 事实错误风险 | 热点新闻可能存在不实或单一来源 | 多源核验，保留来源，人工审核 |
| 品牌口径风险 | AI 内容可能与公司表达不一致 | 引入品牌规范和禁用词，人工确认 |
| 版权风险 | 直接搬运新闻、图片、视频可能侵权 | 只做摘要和再创作，保留来源 |
| 内容质量风险 | AI 生成内容可能模板化 | 引入历史优质内容和平台风格模板 |
| 技术集成风险 | 抖音/小红书发布接口不稳定或不可用 | 第一版以发布包为主，不依赖真实发布接口 |

## 12. CTO 需要关注的决策点

### 12.1 第一版边界

建议确认：第一版是否以“发布包 + 人工审核”为正式目标，而不是直接全自动发布。

### 12.2 技术路线

建议确认：第一版先做本地 POC，暂不引入复杂前端、数据库和队列。

### 12.3 平台策略

建议确认：微信公众号可以优先研究官方草稿接口；抖音、小红书第一版先生成发布包。

### 12.4 风险策略

建议确认：自动评论、点赞、转发不进入第一版。

## 15. 第一版最终建议

第一版建议聚焦以下能力：

```text
热点监控 + 公司相关性判断 + 多源事实摘要 + 三平台图文/视频内容包 + 风险提示 + 发布包输出
```

第一版不建议聚焦以下能力：

```text
全自动发布 + 自动点赞 + 自动评论 + 自动转发 + 多账号矩阵控制
```

该路线能更快验证业务价值，也更容易满足合规、审核、迁移和后续扩展要求。
