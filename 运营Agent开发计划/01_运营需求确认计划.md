# 阶段 0：运营需求确认计划

## 1. 阶段目标

本阶段目标是把运营侧的原始需求转化为可开发、可验收、可控制风险的第一版需求。

重点不是问运营想要哪些功能，而是确认：

```text
她现在怎么做、哪里最痛、第一版最想解决什么、哪些动作必须人工审核。
```

## 2. 需要确认的核心问题

第一次沟通只问少量关键问题：

```text
1. 你现在做自媒体运营，最想被自动化解决的 1-2 个问题是什么？
2. 你现在从发现选题到发布内容，通常怎么做？
3. 哪一步最耗时间、最烦、最容易出错？
4. 你说的自动管理自媒体，最想先自动化哪一段？
5. 如果第一版只能做一个闭环，你希望是什么？
6. 你希望系统直接发布，还是先生成草稿/发布包，由你审核后发布？
7. 你提到自动评论、点赞、转发，背后的业务目的是什么？
8. 第一版上线后，你怎么判断它有用？
```

## 3. 需要带回的结论

沟通结束后，需要沉淀以下内容：

```text
运营核心目标：
当前人工流程：
最大痛点：
平台优先级：
第一版希望实现的闭环：
是否接受人工审核：
是否需要自动发布：
评论/点赞/转发背后的真实目的：
成功标准：
待确认问题：
```

## 4. 运营需求文档结构

输出文档建议包含：

```text
1. 需求背景
2. 运营目标
3. 当前人工流程
4. 当前痛点
5. 第一版需求范围
6. 平台需求
7. 热点监控需求
8. 内容生成需求
9. 发布与审核需求
10. 评论互动需求
11. 第一版不做范围
12. 待确认问题
13. 第一版结论
```

## 5. 第一版需求建议收口

建议将运营需求收口为：

```text
第一版优先做热点监控、相关性判断、多源摘要、三平台内容包和风险提示。
发布环节先生成发布包，由运营人工审核后发布。
评论互动第一版先明确真实目的，暂不做自动点赞、转发和批量评论。
```

## 6. 本阶段交付物

- 运营需求确认记录
- 第一版需求范围说明
- 第一版不做范围说明
- 运营确认后的优先级列表
- 待 CTO 确认的问题清单
